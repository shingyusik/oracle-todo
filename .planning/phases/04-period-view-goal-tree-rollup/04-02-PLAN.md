<objective>
Add the D-10 SQL-pushdown load path: a NEW `load_period_subtree` repository method in `repo.rs` using a SQLite `WITH RECURSIVE` CTE that returns the flat working set (root goals + descendant goals + their tasks) in one indexed query, exposed via a new `TodoRepository` trait method, and wired into the Persistent arm of `period_view`. The CTE applies the exact D-07 visibility predicate chosen in Plan 01 so the Persistent and InMemory stores produce identical working sets.

Purpose: This is the performance half of the phase (D-10) — the first index-backed read in the engine. It exercises Phase 1 indexes (`idx_items_type_horizon_scheduled` seed, `idx_items_parent_id` recursion) instead of the `list_items` full-table scan, while leaving the global `list_items` scan untouched (D-10 scope fence). The in-memory `assemble()` + SC3 safety walk from Plan 01 is unchanged — only the loader differs per store, which is what makes the Plan 03 parity test meaningful.
Output: `load_period_subtree` on the `TodoRepository` trait (ports.rs) + its `SqliteTodoRepository` implementation (repo.rs) + the Persistent branch of `period_view` (queries.rs) replacing the Plan 01 stub.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| service → SQLite (`load_period_subtree`) | `horizon`/`period_key` cross into a SQL statement — the one injection-relevant boundary in this phase. Both are bound as parameters (`?1`/`?2`), never interpolated. `horizon` originates from a typed `Horizon` enum; `period_key` is a normalized canonical date string. |
| recursive CTE → legacy `items` rows | Untrusted legacy data (possible `parent_id` cycles/orphans) is read; the CTE must terminate regardless. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04 | Tampering (SQL injection) | `load_period_subtree` CTE params | mitigate | `params![horizon, period_key]` parameterized binding only; column list is the fixed internal `item_select_sql` constant; status literals are a fixed internal allowlist — NEVER `format!` user/derived values into the SQL (V5.3). |
| T-04-05 | Denial of Service | recursive CTE over cyclic legacy `parent_id` | mitigate | `UNION` (not `UNION ALL`) dedups at the SQL level so back-edges add nothing and the recursion terminates on a finite reachable set; the in-memory `assemble` visited-set + `MAX_GOAL_DEPTH` (Plan 01) remains the locked SC3 guard on top. |
| T-04-06 | Denial of Service | pathological wide subtree | accept | Working set is bounded to the reachable `parent_id` descendants of one period's roots (not the whole table); personal/local single-user workload (CONCERNS.md scaling note). |
| T-04-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed this phase (RESEARCH Package Legitimacy Audit: none added; `rusqlite`/`time`/`serde`/`tempfile` pre-existing and vetted). No `[ASSUMED]`/`[SUS]`/`[SLOP]` packages → no install checkpoint required. |
