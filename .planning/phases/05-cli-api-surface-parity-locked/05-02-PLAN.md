<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → axum router | Untrusted JSON bodies (`/goals/propose`, `PATCH /items/:id`) and query strings (`/views/*`) cross into the process. |
| API handler → TodoService | Handlers call service methods via `with_service`; ALL validation, state-machine, audit, and view logic lives behind this boundary (CORE-01/CORE-03). Handlers set no status and re-implement no view logic. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-API-01 | Tampering | `?horizon=` / `?date=` / `?from=&to=` / `?period=` query params; goal `horizon`/`scheduled` body | mitigate | Forward raw strings to the service; `Horizon::from_str`, `validate_goal_anchor`, and `parse_day` reject junk with `TodoError::Validation` ⇒ HTTP 400 `{"detail": ..}` via `ApiError::into_response`. Only `view_period`/`propose_goal` touch `Horizon::from_str`, mapping `Err(String)` to `TodoError::Validation` (same 400). |
| T-05-API-02 | Elevation of Privilege | `POST /goals/propose` actor/status (approval-gating bypass from API — SC4) | mitigate | Handler NEVER sets status; `parse_actor_or_default(body.actor.as_deref())?` defaults `Agent`, so an agent-created goal starts `proposed`. There is no field or path to create a goal `active` directly from the API; the service state machine owns the status. 05-03 asserts an API-created goal cannot start `active`. |
| T-05-API-03 | Tampering | SQL injection via view/link params | accept | All persistence is parameterized `rusqlite` binds (verified prior phases); this phase adds NO SQL and no string interpolation into queries. Handlers pass strings to already-parameterized service/repo code. |
| T-05-API-04 | Denial of Service | oversized/malformed JSON body; cyclic/over-depth `parent_id` in `/views/period` | accept | Malformed JSON ⇒ `JsonRejection` → `validation_rejection` ⇒ 400 (existing pattern reused). Cyclic/over-depth recursion already mitigated in `queries.rs` (visited set + `MAX_GOAL_DEPTH`, `anomaly_count`, never errors); this plan only exposes the existing read, adding no new traversal. |
| T-05-API-05 | Tampering | `PATCH /items/:id` parent_id to a non-Goal / terminal parent | mitigate | Routed through `update_item` → `ensure_relation(.., Goal, ..)`; non-Goal/terminal ⇒ Policy error ⇒ HTTP 400. Handler adds no bespoke validation (D-09). The Pitfall-1 fix ensures the field is actually wired (not silently dropped to `None`). |
| T-05-API-SC | Tampering | package installs | accept | This phase installs NO new packages (RESEARCH §Standard Stack: "no new dependencies"; all axum/serde/test crates already pinned). No Package Legitimacy Audit required; no install task exists. |
</threat_model>
