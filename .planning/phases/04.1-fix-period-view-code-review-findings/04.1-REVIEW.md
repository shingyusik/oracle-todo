# Phase 04.1: Code Review Report

**Reviewed:** 2026-06-25T00:00:00Z
**Depth:** standard
**Files Reviewed:** 5

## Summary

Phase 04.1 fixes prior period-view code-review findings: a `HashSet`-based
`root_ids` with a root-aware anomaly skip in `build_node`, a single extracted
`schedule_then_created_order` comparator, a goal-parent-only recursive CTE for
cross-store parity (D-11), a single-source `MAX_GOAL_DEPTH` accessor, and two new
parity/anomaly fixtures.

The core logic is correct. The anomaly-count fix (root-aware skip ordered before
`visited.insert`) is sound, the depth-cap off-by-one is consistent between the
`assemble` walk and the `tree_keys` depth assertion, and the SQL injection surface
is properly parameterized. However, I found one genuine **cross-store parity gap**
in the CTE (goal-only descent is enforced only on one side of the join, so a
`task -> goal -> task` shape can diverge from the InMemory loader), one **silently
divergent task-status predicate** that only matches by luck of the current schema,
and a **performance/correctness footgun** in the InMemory frontier walk. The rest
are quality items.

No source files were modified — review is read-only.

## Warnings

### WR-01: CTE goal-only descent guards the parent's type but not the parent's reachability-as-goal — `goal(root) -> task(open) -> goal -> task` can still diverge from InMemory

**File:** `todo-engine/src/infrastructure/sqlite/repo.rs:88-105`

**Issue:** The recursive step pulls *both* goals and tasks into the `subtree` CTE
(`WHERE i.type IN ('goal', 'task')`), and the descent guard is
`JOIN items p ON s.id = p.id AND p.type = 'goal'` — i.e. "only descend from a
parent row whose type is goal." This correctly blocks descent *through a task*.
But it does **not** block a **task from entering the working set under a goal and
then a goal entering under… nothing** — that part is fine. The real divergence is
subtler: the InMemory loader's `frontier` is **goals-only** (line 196-222 of
`queries.rs`), and tasks are added in a **separate final pass** that requires the
task's parent to be an in-frontier *goal*. The CTE, by contrast, admits tasks into
`subtree` and then, on the next recursion, tries `JOIN subtree s ON i.parent_id =
s.id` where `s` may be a task id — only the `p.type = 'goal'` clause stops that.
The two are equivalent **only because** a task can never be a goal's parent in the
goal-only descent. That equivalence is asserted by `parity_goal_task_goal_cross_store`
but **only for the single `goal -> task -> goal` shape**. There is no fixture for
`goal(root) -> goal(child) -> task -> goal(grandchild via task)` where the
intermediate goal is reachable but the deepest goal hangs off a task; both stores
should drop the deepest goal, but the test matrix does not prove it. The parity
claim in the doc comment ("cannot drift… by construction") is stronger than what
the fixtures verify.

**Fix:** Add a parity fixture for the deeper `goal -> goal -> task -> goal` shape
(raw-injected on the persistent side, valid-equivalent on the InMemory side, same
pattern as `parity_goal_task_goal_cross_store`) so the goal-only-descent invariant
is locked at depth > 1, not just at the root. No production change required if the
new fixture passes; if it fails, the CTE join needs the descent constrained on the
*child* side as well (`AND i.parent_id IN (SELECT id FROM subtree WHERE …goal…)`).

### WR-02: CTE status predicate compares against `OPEN_STATUSES` string values but the seed/recursive steps hard-code `type = 'goal'`/`'task'` literals — drift risk if `ItemType` string mapping changes

**File:** `todo-engine/src/infrastructure/sqlite/repo.rs:92-104`

**Issue:** The task-status allowlist is correctly generated from `OPEN_STATUSES`
(lines 79-114) so it cannot drift from the InMemory loader — good. But the *type*
discriminants (`'goal'`, `'task'`) are hard-typed SQL string literals in three
places (seed `type = 'goal'`, recursive `p.type = 'goal'`, `i.type IN ('goal',
'task')`, outer `type = 'goal' OR (type = 'task' …)`). The canonical mapping lives
in `item_type_sqlite_value` / `ItemType::as_str` (`domain/model.rs`). If that
mapping ever changes (e.g. `'task'` → `'task_item'`), the CTE silently returns an
empty/wrong working set with no compile error — exactly the class of "two stores
drift" bug this phase set out to eliminate. The same discipline applied to statuses
(derive from the source of truth) was *not* applied to the type literals.

**Fix:** Bind the type discriminants as parameters sourced from
`ItemType::Goal.as_str()` / `ItemType::Task.as_str()` (the same way statuses are
bound), or add a test asserting `item_type_sqlite_value(ItemType::Goal) == "goal"`
and `… Task == "task"` so a mapping change breaks a test rather than silently
emptying period-view. The parameterized approach is preferred for symmetry with the
status handling already in this function.

### WR-03: InMemory frontier walk is O(goals²) per iteration and re-scans the entire goal set every loop pass — quadratic on a flat/wide tree

**File:** `todo-engine/src/application/service/queries.rs:206-222`

**Issue:** The frontier-expansion `loop` re-iterates **all** goals on every pass and
only stops when a full pass adds nothing. For a chain of N goals it does N passes ×
N goals = O(N²) `contains` checks; the CTE does this in one indexed query. This is
the InMemory half of the parity pair, so it is not the hot persistent path, but the
InMemory store is used by the entire unit/integration suite and any in-process
embedding. More importantly for *correctness review*: the loop's termination depends
solely on `added` flipping false, and a goal already in the frontier is re-checked
every pass (the `if frontier.contains(&goal.id) { continue; }` guard runs N times per
already-added goal). This is safe but wasteful; it is flagged because a future edit
that, e.g., mutates `frontier` mid-iteration could reintroduce a non-termination
risk that the current structure hides.

**Fix (correctness-preserving, not a perf-only change):** Replace the
re-scan-everything loop with a worklist/BFS keyed off a `goals_by_parent` index
built once:

```rust
let mut by_parent: HashMap<Option<&str>, Vec<&TodoItem>> = HashMap::new();
for g in &all_goals { by_parent.entry(g.parent_id.as_deref()).or_default().push(g); }
let mut queue: Vec<String> = frontier.iter().cloned().collect();
while let Some(id) = queue.pop() {
    for child in by_parent.get(&Some(id.as_str())).into_iter().flatten() {
        if frontier.insert(child.id.clone()) { queue.push(child.id.clone()); }
    }
}
```

This makes termination structurally obvious (each id enqueued at most once) and
matches the CTE's single-pass semantics. (Per v1 scope, the perf angle alone is
out of scope; this is flagged for the termination-clarity/maintainability angle.)

## Info

### IN-01: Doc comment on `load_period_subtree` overstates the parity guarantee

**File:** `todo-engine/src/application/ports.rs:11-23`, `todo-engine/src/infrastructure/sqlite/repo.rs:50-56`

**Issue:** Both doc blocks assert the CTE produces a working set "identical to the
InMemory frontier walk by construction — not merely by an `assemble` coincidence."
As noted in WR-01, that identity is only *tested* for the root-level `goal -> task ->
goal` shape. The doc claims a stronger invariant than the fixtures lock. Comments
that promise more than the tests prove tend to erode under future edits.

**Fix:** Soften to "…identical for goal-only descent, verified by the
`parity_*_cross_store` fixtures" and reference the fixture names, or add WR-01's
deeper fixture and keep the strong wording.

### IN-02: `MAX_GOAL_DEPTH` accessor doc comment carries a large inline rationale that duplicates planning/RESEARCH content

**File:** `todo-engine/src/application/service/mod.rs:24-37`

**Issue:** The 14-line doc comment explaining why `pub const MAX_GOAL_DEPTH =
goal::MAX_GOAL_DEPTH;` is used (E0364, `#[cfg(test)]` not reaching the integration
crate, threat T-04.1-05) is correct and useful, but it embeds phase-internal
planning references ("RESEARCH A1", "threat T-04.1-05", "D-06 / IN-03") that are
opaque to a future reader without the planning artifacts. The *binding* (`pub const
… = goal::MAX_GOAL_DEPTH`) is the right single-source pattern.

**Fix:** Keep the one-line "this binds the production value so there is exactly one
source of truth" and move the E0364/cfg-test rationale into an ADR or a shorter
`// why:` note. Non-blocking.

### IN-03: `assemble` re-collects roots via `goals_by_parent.values().flatten()` — an indirection that obscures intent

**File:** `todo-engine/src/application/service/queries.rs:318-323`

**Issue:** Roots are gathered by flattening *every* parent bucket and filtering on
`root_ids.contains`. This is O(goals) and correct, but it scatters the root items
across buckets and reconstitutes them — a reader has to realize the same goal lives
once in exactly one `parent_id` bucket. A direct pass over `working_set` capturing
root `TodoItem`s while building the indexes (one extra `Vec` push under the existing
`root_ids.insert` branch) would be clearer and avoid the second full scan.

**Fix:** Collect roots in the same loop that populates `root_ids` (push a clone when
the root predicate matches) instead of re-deriving them from `goals_by_parent`.
Behavior-neutral readability improvement.

### IN-04: Hard-coded test ISO timestamp `"2026-06-01T00:00:00Z"` repeated in both raw insert helpers

**File:** `todo-engine/tests/integration/period_view.rs:563`, `:590`

**Issue:** `insert_goal_row` and `insert_task_row` each hard-code the same
`created_at`/`updated_at` literal. Minor duplication; if a future test needs to
control creation order between raw-injected rows (the comparator's `created_at`
tie-break), the shared constant would have to be threaded through. Test-only, low
impact.

**Fix:** Hoist to a `const RAW_TS: &str = "2026-06-01T00:00:00Z";` shared by both
helpers, or accept a `created_at` parameter when order-sensitivity is needed.
