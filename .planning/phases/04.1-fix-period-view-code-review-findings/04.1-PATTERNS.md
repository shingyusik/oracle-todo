### `todo-engine/src/infrastructure/sqlite/repo.rs` (infrastructure, read CTE) — D-01, D-03

**Analog:** the existing CTE being modified, repo.rs:81-94. Edit the recursive arm in place.

**Current CTE — the thing being changed** (repo.rs:81-94, VERIFIED verbatim):
```rust
let suffix = format!(
    "WHERE id IN (
         WITH RECURSIVE subtree(id) AS (
             SELECT id FROM items
             WHERE type = 'goal' AND horizon = ?1 AND scheduled = ?2
             UNION
             SELECT i.id FROM items i
             JOIN subtree s ON i.parent_id = s.id
             WHERE i.type IN ('goal', 'task')
         )
         SELECT id FROM subtree
     )
     AND (type = 'goal' OR (type = 'task' AND status IN ({status_placeholders})))"
);
```

**Pattern to apply (D-01):** add one self-join to the recursive arm so descent only
follows GOAL parents. Insert the `JOIN items p ...` line and a clarifying comment (D-03):
```rust
             SELECT i.id FROM items i
             JOIN subtree s ON i.parent_id = s.id
             -- D-11: only descend through goal parents so the CTE working set
             -- matches the InMemory frontier (goal->goal only). A goal whose
             -- parent is a task is unreachable in both stores.
             JOIN items p ON s.id = p.id AND p.type = 'goal'
             WHERE i.type IN ('goal', 'task')
```

**Critical correctness notes (from RESEARCH):**
- The JOIN tests the PARENT row's type (`s.id`), NOT the child `i`. Writing
  `ON i.id = p.id` is the Pitfall-3 bug.
- `WHERE i.type IN ('goal', 'task')` stays — direct-under-goal tasks must still load.
- NO new bound parameter. `?1`/`?2`/`?3..` binding order and `status_placeholders`
  generation (repo.rs:69-75, 98-103) are unchanged. The added predicate is a static
  literal — no value interpolation (V5 unchanged).
- Keep the deduplicating `UNION` (not `UNION ALL`) — it is the SQL-level cycle guard.

**D-03 doc:** the doc block at repo.rs:43-63 already claims "the two stores cannot
drift" / D-07 parity — under D-01 that becomes literally true; keep the wording. Add
only the single inline comment above. Do NOT weaken the doc (no Path B).

#### D-04 + IN-01: HashSet root_ids + anomaly skip

`HashSet` is ALREADY imported (queries.rs:1: `use std::collections::{HashMap, HashSet};`).

Change site 1 — declaration (queries.rs:291):
```rust
let mut root_ids: Vec<String> = Vec::new();   // -> HashSet<String>::new()
```
Change site 2 — population (queries.rs:298): `root_ids.push(...)` -> `root_ids.insert(...)`.
Change site 3 — membership filter (queries.rs:316): `.filter(|goal| root_ids.contains(&goal.id))`
is UNCHANGED (`HashSet::contains` same signature; this also fixes the IN-01 O(roots×goals) scan).

**Current anomaly branch — the thing being changed** (queries.rs:367-382, VERIFIED):
```rust
let mut child_goals = Vec::new();
for child in children {
    if depth + 1 > MAX_GOAL_DEPTH || !visited.insert(child.id.clone()) {
        // Over-depth or cycle (re-visit): sever this branch, count it, never Err.
        *anomaly_count += 1;
        continue;
    }
    child_goals.push(build_node(
        child,
        goals_by_parent,
        tasks_by_parent,
        visited,
        depth + 1,
        anomaly_count,
    ));
}
```

**Pattern to apply (D-04):** insert the root check FIRST in the loop body (Pitfall-2:
must precede `visited.insert`), and thread `root_ids: &HashSet<String>` through
`build_node`'s signature (queries.rs:348-355), the `assemble` call (queries.rs:328-340),
and the recursive self-call (queries.rs:374-381):
```rust
for child in children {
    if root_ids.contains(&child.id) {
        continue; // already emitted as a top-level sibling root (D-02) — NOT an anomaly
    }
    if depth + 1 > MAX_GOAL_DEPTH || !visited.insert(child.id.clone()) {
        *anomaly_count += 1;
        continue;
    }
    child_goals.push(build_node(
        child, goals_by_parent, tasks_by_parent, root_ids, visited, depth + 1, anomaly_count,
    ));
}
```

**Borrow caution (Pitfall-1):** `roots` is built by *borrowing* `root_ids` (`.contains`
at queries.rs:316) — `root_ids` is not moved. Keep it alive (do not shadow/drop) and pass
`&root_ids` into the `build_node` loop at queries.rs:328. Compiler error to watch for:
"borrow of moved value: `root_ids`".

#### D-05 + IN-02: shared comparator extraction

**Analog:** the two byte-identical comparator closures.

Current `sort_date_view` (queries.rs:251-261):
```rust
fn sort_date_view(items: &mut [TodoItem]) {
    items.sort_by(|left, right| {
        let ka = iso_day(left.scheduled.as_deref());
        let kb = iso_day(right.scheduled.as_deref());
        ka.is_none()
            .cmp(&kb.is_none())
            .then_with(|| ka.cmp(&kb))
            .then_with(|| left.created_at.cmp(&right.created_at))
            .then_with(|| left.id.cmp(&right.id))
    });
}
```
`sort_child_goals` (queries.rs:393-403) has a CHARACTER-IDENTICAL closure body (diffed).

**Pattern to apply (D-05):** introduce ONE free `fn` helper (mirror the existing free-fn
style of `iso_day`/`format_iso_day` in this module) and have both `sort_*` delegate:
```rust
/// D-06/D-08 order: iso_day(scheduled) asc with None (unscheduled) LAST,
/// then created_at, then id. Single source for both sort_date_view and sort_child_goals.
fn schedule_then_created_order(left: &TodoItem, right: &TodoItem) -> std::cmp::Ordering {
    let ka = iso_day(left.scheduled.as_deref());
    let kb = iso_day(right.scheduled.as_deref());
    ka.is_none()
        .cmp(&kb.is_none())
        .then_with(|| ka.cmp(&kb))
        .then_with(|| left.created_at.cmp(&right.created_at))
        .then_with(|| left.id.cmp(&right.id))
}

fn sort_date_view(items: &mut [TodoItem]) { items.sort_by(schedule_then_created_order); }
fn sort_child_goals(goals: &mut [TodoItem]) { goals.sort_by(schedule_then_created_order); }
```
KEEP both wrapper fns — call sites reference them by name (queries.rs:319, 358, 365 and
the date_view path). Pure refactor: identical ordering, identical output. RESEARCH
recommends doing this FIRST (lowest risk). `fn` item coerces to `sort_by`'s `fn(&T,&T)->Ordering`.

```rust
// Mirror of insert_goal_row (period_view.rs:474-497) with type='task'. Open status
// 'active' so the D-07 open-task predicate keeps T visible under its parent goal.
fn insert_task_row(
    conn: &rusqlite::Connection,
    id: &str,
    title: &str,
    scheduled: Option<&str>,
    parent_id: Option<&str>,
) {
    conn.execute(
        "INSERT INTO items
            (id, type, title, status, proposed_by, created_at, updated_at, scheduled, parent_id)
         VALUES (?1, 'task', ?2, 'active', 'user', ?3, ?3, ?4, ?5)",
        rusqlite::params![id, title, "2026-06-01T00:00:00Z", scheduled, parent_id],
    )
    .expect("raw task insert");
}
```
(Tasks have no `horizon` requirement for the CTE — only the goal seed must match the period.)

**Fixture construction (insert PARENT-FIRST to satisfy the FK — Pitfall-4):**

**Apply to:** D-07. Compare `tree_keys` (title, depth, kind) sequences, NEVER raw ids
(in-memory uses `goal_000001`, persistent uses UUIDs).

predicate + the in-memory `open_tasks` filter). Referenced, NOT changed. Confirms the
"both stores read from one predicate" pattern that D-01 extends to the working-set frontier.
