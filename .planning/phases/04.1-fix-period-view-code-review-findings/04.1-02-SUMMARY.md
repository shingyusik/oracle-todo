# Phase 04.1 Plan 02: Constrain period-view CTE to goal-parent descent Summary

Made the cross-store parity contract (D-11) true by construction: the SQLite recursive CTE in `repo.rs` now descends only through *goal* parents, so the persistent working set matches the InMemory frontier walk exactly — closing review findings WR-01 (CTE descended through task parents too) and WR-03 (over-stated parity docs).

## What Was Built

**Task 1 — CTE goal-parent descent (D-01 / WR-01), repo.rs**
Added one self-join to the recursive arm of the `WITH RECURSIVE subtree(id)` CTE:

```sql
SELECT i.id FROM items i
JOIN subtree s ON i.parent_id = s.id
-- D-11: only descend through goal parents ...
JOIN items p ON s.id = p.id AND p.type = 'goal'
WHERE i.type IN ('goal', 'task')
```

The new JOIN tests the **parent** row already in the subtree (`s.id = p.id`), not the child `i` (Pitfall-3 avoided). A child is now pulled only when its parent is a goal. For `goal G1(root) -> task T -> goal G2`, the CTE set becomes `{G1, T}` — G2 is dropped because its parent (T) is a task — identical to the InMemory frontier walk in `queries.rs` (goals seeded into the frontier and expanded; tasks appended, never expanded). A single clarifying comment naming D-11 sits directly above the JOIN (D-03).

Preserved exactly:
- `WHERE i.type IN ('goal', 'task')` — direct-under-goal tasks still load.
- The deduplicating `UNION` (not `UNION ALL`) — the SQL-level cycle guard.
- `?1`/`?2`/`?3..` bind order and `status_placeholders` generation — the added predicate is a static literal, no value interpolated (no new bound parameter; V5/T-04.1-02 unchanged).

**Task 2 — parity docs accurate under D-01 (D-03 / WR-03), ports.rs + repo.rs**
Doc-only. Kept the "same flat working set" / "two stores cannot drift" / "identical tree shape (D-11)" wording — now literally true — and tightened each to cite the goal-only CTE descent (D-01) as the enforcement mechanism rather than an `assemble` coincidence. No claim was weakened to "identical rendered tree" (the rejected Path B).

## How It Works

The persistent loader and the InMemory loader both feed the same store-agnostic `assemble()` walk (D-11). Before this plan they happened to render the same tree only by an `assemble` implementation detail (it descends goal->goal, so a goal-under-task was never rendered regardless). The documented "same flat working set" invariant was therefore *false* on adversarial `goal -> task -> goal` linkage: the old CTE pulled G2 into the persistent working set while the InMemory frontier never reached it. This plan makes the invariant enforced in SQL — both working sets are now `{G1, T}` on that shape, so the stores cannot diverge upstream of `assemble`.
