## Finding-by-Decision Research

### D-01 + D-03: CTE goal-only descent (WR-01, WR-03)

**Current SQL (repo.rs:81-94), verified verbatim:**

```sql
WHERE id IN (
    WITH RECURSIVE subtree(id) AS (
        SELECT id FROM items
        WHERE type = 'goal' AND horizon = ?1 AND scheduled = ?2
        UNION
        SELECT i.id FROM items i
        JOIN subtree s ON i.parent_id = s.id
        WHERE i.type IN ('goal', 'task')
    )
    SELECT id FROM subtree
)
AND (type = 'goal' OR (type = 'task' AND status IN ({status_placeholders})))
```

The recursive arm pulls **any** child (goal or task) whose parent is in `subtree`. For
`goal G1(root) → task T → goal G2`: seed `{G1}` → T joins (parent G1 in subtree) → G2 joins
(parent T in subtree). CTE set = `{G1, T, G2}`. The in-memory walk only ever seeds *goals* into
`frontier` (queries.rs:206-222 iterates `all_goals`; tasks are appended afterward at 230-234 and
never expanded), so G2 (parent = task T) is unreachable → in-memory set = `{G1, T}`. **The flat
sets differ — this is WR-01.** `[VERIFIED: codebase repo.rs:81-94, queries.rs:206-234]`

**Minimal correct fix (the JOIN approach from WR-01 / D-01):** constrain the recursive arm so a
child is only pulled when its parent row is a *goal*:

```sql
SELECT i.id FROM items i
JOIN subtree s ON i.parent_id = s.id
JOIN items p ON s.id = p.id AND p.type = 'goal'   -- D-11: only descend through goal parents
WHERE i.type IN ('goal', 'task')
```

Trace under the fix: seed `{G1}`. Recursive step: candidate T has parent G1; `p` = G1 row,
`p.type = 'goal'` ✓ → T joins. Next iteration: candidate G2 has parent T; `p` = T row,
`p.type = 'task'` ✗ → G2 dropped. CTE set = `{G1, T}` — **identical to the in-memory set.**
`[VERIFIED: codebase + SQL trace]`

**SQL correctness notes (rusqlite / SQLite CTE):**
- The new `JOIN items p ON s.id = p.id` joins the *parent* row (the row already in `subtree`,
  identified by `s.id`) and tests its `type`. This is a standard self-join inside a recursive
  CTE; SQLite supports multiple JOINs in the recursive `SELECT`. `[CITED: sqlite.org/lang_with.html]`
- `subtree(id)` carries only `id`, so to test the parent's type we must re-join `items p ON s.id = p.id`.
  An equivalent alternative (Claude's discretion per CONTEXT) is to **widen the CTE to carry the
  parent's type forward** — e.g. `subtree(id, type)` selecting `i.type` and filtering
  `WHERE s.type = 'goal'`. The extra-JOIN form is simpler and matches the WR-01 suggested fix
  exactly; recommend it unless a planner prefers the carried-column form. `[ASSUMED — both compile; extra-JOIN is the reviewed form]`
- **No new bound parameter** is introduced; `?1`/`?2`/`?3..` binding order is unchanged. The
  status-placeholder generation (repo.rs:69-75) and param push (98-103) stay exactly as-is.
  `[VERIFIED: codebase repo.rs:69-103]`
- The `UNION` (deduplicating, not `UNION ALL`) is retained — it remains the SQL-level cycle guard.
  The new JOIN narrows *which* children are pulled, not the dedup behavior. `[VERIFIED: codebase repo.rs:83-90]`
- `WHERE i.type IN ('goal', 'task')` in the recursive arm can stay; with the goal-parent JOIN the
  task `T` is still pulled (its parent G1 is a goal), which is correct — direct-under-goal tasks must
  load. Only *goals whose parent is a task* are now excluded. `[VERIFIED: SQL trace]`

**Rendered-output-unchanged proof (D-02):** `assemble` descends goals **only** via
`goals_by_parent.get(&Some(goal.id))` (queries.rs:362) and attaches tasks via
`tasks_by_parent.get(&goal.id)` (queries.rs:357). G2's parent is task T, and `goals_by_parent`
is keyed by `parent_id`, so G2 sits under key `Some(T)` which is never queried (T is a task, never
a node). G2 was therefore **never a rendered node** even with the old CTE — dropping it from the
working set is invisible to every caller. `[VERIFIED: codebase queries.rs:289-389]`

**D-03 doc fix:** Update the over-stated parity claims in `ports.rs:11-18` ("produce identical tree
shape (D-11)") and `repo.rs:43-63` so they remain accurate — under D-01 the "same flat working set"
claim becomes literally true. Add **one** clarifying comment at the new recursive-step JOIN stating
the goal-parent constraint exists to enforce D-11. Per CONTEXT D-03, **keep** the existing "produce
the SAME flat working set" wording (do not take the Path-B doc-weakening route). `[VERIFIED: CONTEXT D-03]`

### D-04 + IN-01: anomaly_count over-count + HashSet root_ids (WR-02)

**Current `root_ids` flow (verified):**
- Declared `Vec<String>` at queries.rs:291.
- Populated via `root_ids.push(...)` at queries.rs:298 (inside the working-set partition loop).
- Membership-tested via `root_ids.contains(&goal.id)` at queries.rs:316 (the O(roots×goals) scan, IN-01).
- **Never indexed, never ordered, never iterated for anything but membership** → a `HashSet<String>`
  is a drop-in. `[VERIFIED: codebase queries.rs:291-318]`

**Change set for D-04 (every call site that must change):**
1. queries.rs:291 — `let mut root_ids: Vec<String> = Vec::new();` → `HashSet<String>`.
   (`HashSet` already imported at queries.rs:1 — `use std::collections::{HashMap, HashSet};`.) `[VERIFIED]`
2. queries.rs:298 — `root_ids.push(item.id.clone());` → `root_ids.insert(item.id.clone());`.
   (`insert` ignores duplicates, which is correct — a root id is added once.) `[VERIFIED]`
3. queries.rs:316 — `.filter(|goal| root_ids.contains(&goal.id))` is **unchanged** (`HashSet::contains`
   has the same signature; this simultaneously fixes IN-01's O(n²) scan). `[VERIFIED]`
4. `build_node` signature (queries.rs:348-355) — add a `root_ids: &HashSet<String>` parameter.
5. The `assemble` call to `build_node` (queries.rs:328-340) — pass `&root_ids`. **Borrow ordering
   caution:** `roots` is currently *moved out of* `root_ids`-derived data at queries.rs:313-318
   (`roots` is built by filtering on `root_ids` then `root_ids` is no longer consumed). Confirm
   `root_ids` is **not moved** before the `build_node` loop — it is only borrowed (`.contains`), so
   passing `&root_ids` into the loop at line 328 is valid. Keep `root_ids` alive (do not shadow/drop
   it) through the `build_node` calls. `[VERIFIED: codebase queries.rs:313-341]`
6. The recursive `build_node` self-call (queries.rs:374-381) — thread `root_ids` through.

**The anomaly-skip logic (the actual fix, per WR-02):** In `build_node`'s child loop
(queries.rs:368-373), add the root check **before** the visited/depth check:

```rust
for child in children {
    if root_ids.contains(&child.id) {
        continue; // already emitted as a top-level sibling root (D-02) — NOT an anomaly
    }
    if depth + 1 > MAX_GOAL_DEPTH || !visited.insert(child.id.clone()) {
        *anomaly_count += 1;
        continue;
    }
    child_goals.push(build_node(child, goals_by_parent, tasks_by_parent, root_ids, visited, depth + 1, anomaly_count));
}
```

**Why this is correct (verified against current code):** Today, every root is pre-marked `visited`
at queries.rs:325-327. When a sibling-root child (R2 whose `parent_id` is R1, both roots) is reached
in R1's `goals_by_parent` bucket, `visited.insert(R2.id)` returns `false` → the branch is severed AND
`*anomaly_count += 1` (queries.rs:369-372). The new `root_ids.contains(&child.id)` branch intercepts
R2 first and `continue`s **without** the bump — R2 stays a top-level sibling (the D-02 intent) and the
count stays clean. Genuine cycles, over-depth, and orphan re-visits (children that are *not* roots)
still fall through to the existing `*anomaly_count += 1`. `[VERIFIED: codebase queries.rs:322-389]`

**Order matters:** the root check must precede `visited.insert` so that a non-root cycle/over-depth
still bumps the count. Placing it first does not double-skip genuine anomalies because only *roots*
short-circuit. `[VERIFIED: logic trace]`

### D-05 + IN-02: shared sort comparator

**Byte-identity confirmed.** `sort_date_view` (queries.rs:251-261) and `sort_child_goals`
(queries.rs:393-403) have **character-identical closure bodies** — both:
```rust
let ka = iso_day(left.scheduled.as_deref());
let kb = iso_day(right.scheduled.as_deref());
ka.is_none().cmp(&kb.is_none())
    .then_with(|| ka.cmp(&kb))
    .then_with(|| left.created_at.cmp(&right.created_at))
    .then_with(|| left.id.cmp(&right.id))
```
`[VERIFIED: codebase queries.rs:251-261 vs 393-403 — diffed]`

**Ordering key (for the doc comment on the new helper):** `iso_day(scheduled)` ascending with
`None` (unscheduled) **last** → `created_at` → `id`. This is the D-06/D-08 deterministic order.

**Recommended extraction (per D-05):** introduce one pair-comparator helper and have both
`sort_*` functions delegate:

```rust
/// D-06/D-08 order: iso_day(scheduled) asc with None last, then created_at, then id.
fn schedule_then_created_order(left: &TodoItem, right: &TodoItem) -> std::cmp::Ordering {
    let ka = iso_day(left.scheduled.as_deref());
    let kb = iso_day(right.scheduled.as_deref());
    ka.is_none().cmp(&kb.is_none())
        .then_with(|| ka.cmp(&kb))
        .then_with(|| left.created_at.cmp(&right.created_at))
        .then_with(|| left.id.cmp(&right.id))
}

fn sort_date_view(items: &mut [TodoItem]) { items.sort_by(schedule_then_created_order); }
fn sort_child_goals(goals: &mut [TodoItem]) { goals.sort_by(schedule_then_created_order); }
```
`std::cmp::Ordering` is referenceable; `sort_by` accepts a `fn(&T, &T) -> Ordering`. Note: passing
`schedule_then_created_order` directly to `sort_by` works because it is a `fn` item coercible to the
closure type. **Keep both wrapper fns** (call sites at queries.rs:91, 106, 319, 358, 365 reference
them by name) to minimize the diff and preserve the existing call surface. `[VERIFIED: call sites grepped]`

This is a pure refactor: identical ordering, identical output. Lowest-risk change — recommend doing
it first to de-risk the diff.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Membership test for root ids | Linear `Vec::contains` scan | `HashSet<String>` (D-04/IN-01) | O(1) vs O(roots×goals); already conceptually a set |
| Duplicate comparator logic | Two byte-identical `sort_by` closures | One `schedule_then_created_order` helper (D-05) | Drift risk if one ordering rule changes |
| Test depth-cap constant | Hand-mirrored `64` in the test (D-06/IN-03) | `pub use goal::MAX_GOAL_DEPTH` re-export | Single source of truth; compiler-enforced sync |
| Goal-only CTE descent | A second post-load filter pass in Rust | Constrain the recursive arm in SQL (D-01) | Keep the working-set producer in one place (the CTE); avoids a divergent Rust filter |
| Building adversarial shapes | Adding service methods that bypass validation | Raw SQLite inserts in the test (D-07/D-08) | The service *must* reject these shapes; tests model legacy/corrupt data via raw rows (documented pattern) |

**Key insight:** Every locked decision *removes* hand-rolled coincidence and replaces it with an
enforced invariant. The phase's value is in making existing guarantees structural, not in adding logic.

## Common Pitfalls

### Pitfall 1: Borrow/move ordering when threading `root_ids` into `build_node`
**What goes wrong:** `roots` is derived by filtering on `root_ids` (queries.rs:313-318); if the planner
moves or shadows `root_ids` before the `build_node` loop, passing `&root_ids` fails to compile.
**How to avoid:** `root_ids` is only *borrowed* (`.contains`) when building `roots` — keep it alive and
pass `&root_ids` into the loop and the recursive self-call. `[VERIFIED: queries.rs:313-341]`
**Warning signs:** "borrow of moved value: `root_ids`" from the compiler.

### Pitfall 2: Anomaly skip placed *after* the visited/depth check
**What goes wrong:** If `root_ids.contains` is checked after `visited.insert`, a sibling root would
still fail `visited.insert` (it was pre-marked at queries.rs:325-327) and bump the count before the
skip is reached.
**How to avoid:** Put `if root_ids.contains(&child.id) { continue; }` **first** in the child loop body.
**Warning signs:** D-08 fixture asserts `anomaly_count == 0` but gets ≥1.

### Pitfall 3: CTE JOIN tests the wrong row's type
**What goes wrong:** Writing `JOIN items p ON i.id = p.id AND p.type = 'goal'` tests the *child's* type
(wrong) instead of the parent's. The constraint must be on the **parent** row (`s.id`), not the candidate `i`.
**How to avoid:** Join `items p ON s.id = p.id AND p.type = 'goal'` — `s` is the already-in-subtree
parent. Verify with the `goal→task→goal` trace: G2 must drop because its parent T is a task.
**Warning signs:** D-07 parity fixture shows G2 present in disk but absent in mem (sets still diverge).

### Pitfall 4: FK violation when raw-inserting goal-under-task / sibling-root
**What goes wrong:** `parent_id REFERENCES items(id)`; inserting a child before its parent fails the FK.
**How to avoid:** Insert parent-first (G1 → T → G2; R1 → R2). Only use `PRAGMA foreign_keys = OFF/ON`
(period_view.rs:562-575) for genuinely dangling references (the orphan case), not here.
**Warning signs:** `FOREIGN KEY constraint failed` from a raw insert.

### Pitfall 5: Raw task fixture uses a terminal status and disappears
**What goes wrong:** The D-07 task T inserted with a terminal status is filtered by the open-task
predicate in both loaders, so the fixture no longer exercises a *visible* task under the goal.
**How to avoid:** Insert T with an open status (`'active'`, matching `insert_goal_row`'s convention).
**Warning signs:** `tree_keys` shows no task row under G1 in either store.

## Open Questions

1. **How to seed the in-memory store with an adversarial `goal→task→goal` shape (D-07)?**
   - What we know: The persistent half uses raw SQLite inserts (FK-bypassing where needed). The
     in-memory store is a `HashMap<String, TodoItem>` populated only through the validating service API,
     which rejects goal-under-task (goal.rs:60-64).
   - What's unclear: Whether to (a) add a narrow test-only constructor/accessor to inject `TodoItem`s
     directly into the in-memory store, or (b) restructure the parity assertion to run the two
     *loaders* (`load_period_subtree_in_memory` vs `load_period_subtree` CTE) over one shared raw
     dataset and compare working sets directly.
   - Recommendation: Prefer (b) if a loader-level comparison is reachable — it tests the exact divergence
     point (working-set construction) without widening the production surface. If only the public
     `period_view` is testable, (a) requires a `#[cfg(test)]` seeding hook. **Decide in planning.**

2. **Does the `WHERE i.type IN ('goal', 'task')` in the recursive arm stay, or simplify to goal-only?**
   - What we know: With the goal-parent JOIN, tasks are still pulled (parent is a goal). Keeping
     `IN ('goal','task')` is correct and minimal.
   - Recommendation: Keep it unchanged — narrowing it risks dropping direct-under-goal tasks. The JOIN
     is the only needed change.
