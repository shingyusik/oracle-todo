# Phase 04.1 Plan 01: queries.rs anomaly_count + comparator fixes Summary

One-liner: De-duplicated the two byte-identical period/date sort comparators into a single
`schedule_then_created_order` free fn, and corrected `anomaly_count` so a structurally valid
D-02 sibling-root config (root parented to another exact-period root) reports `0` instead of
over-counting (WR-02 / IN-01), while migrating `root_ids` to a `HashSet` to kill the
O(roots x goals) membership scan.

## What Was Built

### Task 1 — Shared `schedule_then_created_order` comparator (D-05 / IN-02)
Introduced one free function `fn schedule_then_created_order(left: &TodoItem, right: &TodoItem)
-> std::cmp::Ordering` in `queries.rs`, body identical to the previously-duplicated closure
(`iso_day(scheduled)` asc with `None` LAST, then `created_at`, then `id`). Both `sort_date_view`
and `sort_child_goals` are now one-line delegations `items.sort_by(schedule_then_created_order)`
(an `fn` item coerces to `sort_by`'s `fn(&T,&T)->Ordering`). Both wrapper fns are kept — call
sites reference them by name. Pure refactor: ordering byte-identical. Commit `5a0d5bb`.

### Task 2 — HashSet `root_ids` + root-aware anomaly skip (D-04 / IN-01 / WR-02)
- `root_ids` changed from `Vec<String>` to `HashSet<String>`; population `.push()` -> `.insert()`.
  The membership filter `root_ids.contains(&goal.id)` is unchanged (`HashSet::contains` same
  signature) and now O(1), killing IN-01's O(roots x goals) scan.
- Threaded `root_ids: &HashSet<String>` through `build_node`'s signature, the `assemble` call,
  and the recursive self-call.
- Added `if root_ids.contains(&child.id) { continue; }` as the FIRST statement in `build_node`'s
  child loop — BEFORE the `depth + 1 > MAX_GOAL_DEPTH || !visited.insert(...)` guard — so a
  re-visited child that is itself a period root (D-02 sibling root) is skipped WITHOUT bumping
  `anomaly_count`. Genuine over-depth / non-root re-visits still fall through to the existing
  `*anomaly_count += 1`. Commit `c09ce5c`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Corrected `cycle_is_severed_no_error` fixture to assert the fixed WR-02 behavior**
- **Found during:** Task 2 (period_view suite failed: "cycle must bump anomaly_count (got 0)").
- **Issue:** The existing `cycle_is_severed_no_error` fixture (period_view.rs:511-543) builds its
  "cycle" from TWO goals BOTH anchored to `(month, 2026-06-01)` — i.e. two exact-period matches.
  Under D-04 those are valid D-02 sibling roots, not an anomaly, so the re-visit is now correctly
  intercepted by `root_ids.contains` and `anomaly_count` is `0`. The fixture's old assertion
  (`anomaly_count >= 1`) encoded exactly the WR-02 over-count this plan removes.
- **Topology proof (why this is correct, not a regression):** With strict single-parent
  `parent_id`, a cycle is only ever loaded into the working set when at least one cycle node is a
  period root (a non-root cycle is unreachable from any seed). Re-visiting a root is, by D-04
  intent, NOT an anomaly (it is already emitted as a top-level sibling). The genuine
  sever-and-count path (depth/visited) is independently exercised and still green via
  `depth_cap_truncates` / `depth_cap_truncates_persistent`. The sibling-root `== 0` invariant is
  additionally locked by the Plan 03 D-08 fixture.
- **Fix:** Updated the fixture assertion to `assert_eq!(view.anomaly_count, 0, ...)` plus a check
  that both same-period nodes still surface as top-level sibling roots; rewrote the doc comment to
  explain the D-04 reclassification and the topology argument. Kept the store-level cycle so the
  test still proves non-hang (Ok + termination).
- **Files modified:** todo-engine/tests/integration/period_view.rs
- **Commit:** c09ce5c

**Note on plan instruction conflict:** The plan's Task 2 `<behavior>` said "the existing
period_view.rs cycle/orphan/over-depth fixtures stay green." That instruction assumed this
fixture tested a genuinely countable cycle. It did not — it tested two same-period roots, which is
the exact D-02 sibling-root case the plan's own must-haves require to yield `anomaly_count == 0`.
The plan's explicit must-have ("A valid D-02 sibling-root config ... yields anomaly_count == 0")
is authoritative; honoring it necessarily updates this one fixture's assertion. Orphan and
over-depth fixtures stayed green unchanged.
