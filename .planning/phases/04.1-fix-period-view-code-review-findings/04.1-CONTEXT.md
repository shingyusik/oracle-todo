<decisions>
## Implementation Decisions

### Parity contract — WR-01 / WR-03
- **D-01:** Adopt **CTE goal-only descent (Path A)**. Constrain the SQLite
  recursive-CTE step in `repo.rs` so it descends only through *goal* parents
  (e.g. add `JOIN items p ON s.id = p.id AND p.type = 'goal'` to the recursive
  arm). Under this change the InMemory frontier walk and the CTE produce the
  **identical flat working set**: for `goal G1(root) → task T → goal G2`, both
  yield `{G1, T}` (G2 is dropped from the CTE set because its parent is a task).
  The D-11 "the two stores cannot drift" invariant becomes enforced by code, not
  by an `assemble` implementation detail.
- **D-02:** Rendered output is unchanged. `assemble` already descends goal→goal
  only (`goals_by_parent.get(&Some(goal.id))`), so G2 was never a rendered node;
  dropping it from the CTE working set is invisible to callers. Tasks linked
  directly under goals still load (a task's parent is a goal).
- **D-03:** Keep the existing `ports.rs` / `repo.rs` doc wording ("produce the
  SAME flat working set" / D-11) — it becomes literally true under D-01. Add a
  single clarifying comment at the CTE recursive step stating the goal-parent
  JOIN exists to guard D-11. Do **not** take the Path B doc-weakening route.

### Anomaly over-count — WR-02 (+ IN-01)
- **D-04:** Fix the `anomaly_count` over-count on valid D-02 sibling-root nesting
  by threading the root id set into `build_node`. Change `root_ids` from
  `Vec<String>` to `HashSet<String>` (this simultaneously resolves IN-01's
  O(roots × goals) `Vec::contains` scan). In `build_node`, when a re-visited
  child is itself a root (`root_ids.contains(&child.id)`), `continue` **without**
  incrementing `anomaly_count` — it was already emitted as a top-level sibling
  root. Only genuine cycle / over-depth / orphan cases bump the count.

### Minor cleanups — IN-02, IN-03
- **D-05:** IN-02 — extract the byte-identical `sort_date_view` /
  `sort_child_goals` comparator into one helper
  `fn schedule_then_created_order(a, b) -> Ordering`
  (`iso_day(scheduled)` with `None` last → `created_at` → `id`); both call sites
  delegate to it.
- **D-06:** IN-03 — single-source the test's `MAX_GOAL_DEPTH` (currently a
  hand-mirrored `64` in `period_view.rs`). Expose the production cap to the test
  crate via a narrow test-only accessor / `pub` re-export so the bound has one
  source of truth, while respecting the `pub(super)` visibility convention.

### Tests — WR-04 + WR-02 regression
- **D-07:** WR-04 — add a raw-injection **cross-store** parity fixture building
  `goal(root) → task → goal` (via `insert_goal_row` + a raw task insert parented
  to the goal + a raw goal insert parented to the task). Assert
  `tree_keys(mem) == tree_keys(disk)` **and**
  `anomaly_count(mem) == anomaly_count(disk)`. Under D-01 this *enforces* the
  working-set invariant rather than assuming it.
- **D-08:** WR-02 regression — add a fixture for a valid D-02 sibling-root config
  (root R2 whose `parent_id` is another root R1, both exact period matches) and
  assert `anomaly_count == 0`. Guards the over-count fix against regression.
  Store-level coverage is sufficient here; cross-store is not required for this
  fixture (the parity fixture in D-07 already covers cross-store).

### Claude's Discretion
- Exact SQL phrasing of the goal-parent JOIN (extra JOIN vs. correlated
  predicate) and the precise mechanism for the test-only `MAX_GOAL_DEPTH`
  accessor (D-06) are implementation details left to the planner/executor, as
  long as the invariant and visibility convention hold.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `assemble()` / `build_node()` (queries.rs) — store-agnostic tree walk; the
  anomaly fix (D-04) threads `root_ids` through `build_node`'s existing
  recursion, no new traversal.
- `OPEN_STATUSES` (status.rs) — single open-status predicate already shared by
  both loaders; nothing to change, confirms the shared-source pattern.
- `insert_goal_row` + raw insert helpers (period_view.rs tests) — the vehicle for
  the WR-04 raw-injection fixture (D-07) and the WR-02 sibling-root fixture (D-08).
- `parity_in_memory_vs_persistent` (period_view.rs §391-425) — existing
  cross-store parity test pattern the new D-07 fixture extends.

### Established Patterns
- Clean/hexagonal layering: CTE stays in `infrastructure/`, `assemble` is
  store-agnostic in `application/`, `domain/` does no I/O. D-01's SQL change is
  confined to `repo.rs`; D-04/D-05 stay in `queries.rs`.
- `pub(super)` visibility convention — D-06 must expose `MAX_GOAL_DEPTH` narrowly
  (test-only accessor) without widening the production surface broadly.

### Integration Points
- The CTE working set and the InMemory working set converge at the shared
  `assemble()` entrypoint. D-01 makes their *inputs* identical so the existing
  shared walk cannot diverge across stores.

</code_context>
