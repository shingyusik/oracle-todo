<objective>
Land the two `queries.rs` production fixes that make `anomaly_count` correct and remove a
duplicated comparator, without changing rendered `PeriodView` output.

- D-05 / IN-02: extract the two byte-identical sort comparators into one
  `schedule_then_created_order(left, right) -> Ordering` helper; both `sort_*` fns delegate.
- D-04 / IN-01 / WR-02: change `root_ids` from `Vec<String>` to `HashSet<String>` (fixes the
  O(roots x goals) `Vec::contains` scan) and thread it into `build_node` so a re-visited child
  that is itself a root is skipped via `continue` WITHOUT incrementing `anomaly_count`.

Purpose: A structurally valid D-02 sibling-root config (root parented to another root, both
exact period matches) currently reports `anomaly_count >= 1` even though nothing is cyclic,
orphaned, or over-deep. `anomaly_count` is a public `PeriodView` field surfaced to callers
(SC3), so the conflation is observable. This plan corrects the counter only — never the tree.

Output: `queries.rs` with the shared comparator + HashSet root_ids + root-aware anomaly skip.
The D-08 regression fixture that locks this behavior lands in Plan 03 (Wave 2).
</objective>
