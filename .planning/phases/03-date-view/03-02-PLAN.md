    Implement these `#[test]` fns (names are the VALIDATION.md oracle keys):

    1. `range_orders` (SC1 / VIEW-02, also covers D-08): create open tasks with scheduled dates spanning a range and same-day ties differing by created_at/id; call `date_range("2026-06-01", "2026-06-30")`; assert the returned id sequence equals the expected order = scheduled ascending, then `created_at`, then `id`. Include at least two same-day tasks to prove the tie-break.
    2. `unscheduled_never_dropped` (SC2 / VIEW-02): exercise the FULL-SET / agenda path where unscheduled rows are in scope (NOT `date_range`, which by D-03 narrows to in-range scheduled rows and legitimately excludes unscheduled — see the INFO note in this plan). Create N open tasks, M of them with non-ISO/None `scheduled` — cover `None`, the legacy `"today"` sentinel (set via `update_item` scheduled = "today"), and a junk value (e.g. `"not-a-date"`); call the agenda/full-set path under test; assert the returned Vec length includes all M unscheduled rows AND they occupy the tail of the ordering (sorted last). They must NOT be dropped.
    3. `agenda_union_dedup` (SC3 / VIEW-05): create a task with `scheduled == D`, another with `due == D`, and a third with BOTH `scheduled == D AND due == D`; call `agenda("2026-06-23")`; assert all three appear, and the both-match task appears exactly once — `agenda.iter().filter(|i| i.id == both_id).count() == 1`.
    4. `open_only` (D-05): create tasks in `Proposed`, `Approved`, `Active`, and at least one each of the REACHABLE non-open statuses — `Completed` (via `complete`), `Waiting` (via the routine-pause cascade, or skip if a routine fixture is too heavy), and `Paused` (via `pause`). Do NOT attempt a `Someday` task: the state machine exposes no path that produces `ItemStatus::Someday`, so it is not drivable in a test; the `OPEN_STATUSES` allowlist in 03-01 already excludes it by construction (allowlist semantics — anything not in {Proposed, Approved, Active} is excluded, proven by the reachable Completed/Waiting/Paused exclusions). Call a date-view method that should include the open ones by date; assert only the Proposed/Approved/Active ids are returned and the reachable terminal/hidden ones (Completed/Waiting/Paused) are absent.
    5. `no_overdue_roll` (D-06): create an open task scheduled `2026-06-20`; call `agenda("2026-06-23")`; assert the task is ABSENT (the legacy `scheduled <= today` roll is NOT replicated — exact-date only).

    Use the `filter.rs` id-collection assertion idiom for ordered comparisons. Do NOT assert on raw seeded ids in a way that couples to the in-memory id format beyond what the test creates. Cover D-08 within `range_orders` (no separate test needed, per the test map).
