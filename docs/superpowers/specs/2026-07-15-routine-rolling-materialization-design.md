# Routine Rolling Materialization Design

## Goal

Active routines maintain a configured number of future generated tasks without requiring repeated manual materialization.

## Routine Policy

- `future_occurrences` is stored on each routine.
- The default is `7`; valid values are `1..=365`.
- `single_open` maintains one non-terminal generated task regardless of `future_occurrences`.
- `per_occurrence` maintains `future_occurrences` non-terminal generated tasks.
- Past occurrences are never generated.
- Existing generated tasks are never deleted when the target is reduced.
- A reduced target suppresses replenishment until the number of non-terminal generated tasks falls below the target.

## Materialization Flow

### Activation

Activating a routine validates its recurrence rule, stores the active routine, and fills its target from the current date forward.

### Completion

Completing a task records the terminal occurrence as usual. When the task was generated by an active routine, the service fills that routine back to its target beginning after its latest generated occurrence.

Completion of unrelated tasks and completion of tasks whose routine is paused or terminal do not materialize work.

### Manual Materialization

The routine materialization action accepts `future_occurrences`, saves it on the routine, and fills any shortage. Repeating the action is idempotent because `(routine_id, occurrence_key)` remains unique.

Increasing the target creates missing future tasks. Decreasing it keeps existing tasks and creates nothing until the open count falls below the new target.

## Data Model and API

- Add `items.future_occurrences INTEGER NOT NULL DEFAULT 7` through the additive schema initializer.
- Include `future_occurrences` in routine creation, item update, persistence mapping, and serialized item responses.
- Replace materialization request fields `lookahead_days` and `catchup_days` with `future_occurrences`.
- Reject values outside `1..=365` as validation errors.

## User Interface

- Replace `Catchup days` and `Lookahead days` with one `Future occurrences` number input.
- The materialize action saves the target and fills missing tasks.
- Keep the button as an explicit repair or immediate-fill action; normal activation and completion flows are automatic.

## Error and Audit Behavior

- All routine and generated-task writes continue through `TodoService`.
- Generated tasks retain the `materialize_routine_task` audit action.
- Target changes are included in the routine mutation snapshot.
- Unsupported recurrence rules fail before automatic generation begins.
- Uniqueness conflicts during concurrent generation are treated as an already-filled occurrence.

## Verification

- Service tests cover initial fill, completion replenishment, `single_open`, target increase, and non-destructive target reduction.
- API tests cover request validation and returned routine state.
- UI tests cover the single target input and request payload.
- Existing schema migration tests cover the additive default for older databases.
- Full Rust formatting, test, and lint gates pass, followed by a throwaway-data-home smoke run.
